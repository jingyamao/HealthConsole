# 使用 Agent / AI 驱动构建的具体成果

## 一、项目解决的核心痛点

### 痛点 1：医生病历查阅效率低

医生需在多个页面间切换查看患者的病史、检查、诊断等分散数据。系统通过 Agent 自然语言查询，医生直接提问（如「最近一周收了多少高血压患者」），Agent 自动转换为 Prisma 查询并返回格式化答案。

### 痛点 2：病历数据分析依赖人工经验

患者数据分散在 9 个关联表中，医生需手动整合。系统通过 AI 综合分析 Agent，将全量病历格式化为上下文，调用 LLM 进行 7 维度临床评估（风险等级、关键发现、诊断评估、治疗审查、随访建议等），结果持久化保存。

### 痛点 3：医学知识检索与临床决策脱节

通过 RAG 架构，17 篇医学文档向量化存储，AI 对话时自动触发语义检索，将相关文献注入 LLM 回答，实现循证医学问答。

### 痛点 4：结构化病历表单填写繁琐

传统 JSON 自由文本框易填错。系统实现三种可视化输入组件（tags 标签 / kvlist 键值对 / rows 动态行），自动转换为正确 JSON 格式存入数据库。

## 二、核心逻辑流

### 架构：单 Agent 多工具编排（非多 Agent 协作）

医疗场景对延迟敏感，采用中央编排器 + 工具分发的单 Agent 模式：

```
用户输入 → AgentManager → 意图识别(LLM 5路分类) → 工具调度:
  ├── databaseQuery : 3步长链推理 (LLM解析意图→执行查询→LLM格式化)
  ├── medicalChat   : 医学系统提示词 + RAG上下文增强
  ├── vectorSearch  : 4步RAG链 (嵌入→余弦检索→聚合→上下文注入→LLM回答)
  └── listKnowledge : 知识库枚举
```

### 长链推理：Database Query Tool

三步串行 LLM 调用，前一步输出为下一步输入，任意偏差均影响最终结果：

```
用户自然语言
  → LLM Call 1: 生成 JSON 查询计划（目标表、筛选条件、聚合方式）
  → executeQueryPlan(): 映射到 10 张表的 Prisma 查询（无 LLM，确定性执行）
  → LLM Call 2: 原始结果格式化为自然语言 Markdown 答案
```

### RAG 长链：向量检索 + 增强生成

4 步链式处理，实现带源引用的医学问答：

```
用户问题 → 生成查询嵌入 → 余弦相似度检索(阈值 0.7) → 同源聚合
  → 上下文注入 system prompt → medicalChat LLM 生成循证答案
```

### AI 临床分析：长上下文推理

将患者 9 类关联数据格式化为 2000+ 字符结构文本，单次 LLM 调用提取关键信号：

```
全量病历 → formatPatientContext() → LLM(临床专家角色+风险标准)
  → 结构化 JSON { riskLevel, keyFindings, diagnosisAssessment, ... }
  → 持久化到 Patient.aiNotes
```

## 三、技术架构总结

| 维度 | 方案 |
|------|------|
| Agent 模式 | 单 Agent 多工具调度 |
| 推理链长度 | databaseQuery 3 步串行 LLM；RAG 4 步链式 |
| 多 Agent 协作 | 否（单编排器 + 工具分发） |
| LLM 策略 | KIMI（推理/分析）+ MIMO（文档生成含联网搜索） |
| 向量存储 | 自研 PostgreSQL JSONB + JS 余弦相似度 |
| 结构化输出 | Zod schema 约束输入，JSON mode 约束输出 |
