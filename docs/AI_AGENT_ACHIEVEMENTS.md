# 使用 Agent / AI 驱动构建的具体成果

## 一、项目解决的核心痛点

### 痛点 1：医生病历查阅效率低

传统 EMR（电子病历）系统中，医生需要在多个页面间切换查看患者的病史、症状、检查结果、诊断和治疗方案。本系统通过 Agent 的自然语言数据库查询能力，让医生可以直接用自然语言提问（如「最近一周心内科收了多少高血压患者」），Agent 自动将问题转换为 Prisma 查询计划，执行多表联查，并返回格式化答案。

### 痛点 2：病历数据分析依赖人工经验

患者数据分散在 9 个关联表中（病史、症状、体格检查、检查结果、诊断、治疗、病程、财务、医疗团队），医生需要手动整合信息才能形成整体判断。系统通过 AI 综合分析 Agent，将患者全量病历格式化为结构化上下文，调用 LLM 进行 7 维度临床评估（风险等级、关键发现、关注要点、诊断评估、治疗审查、随访建议），并输出结构化 JSON 结果持久化保存。

### 痛点 3：医学知识检索与临床决策脱节

医疗指南和文献知识通常存放在独立的知识库系统中，医生需要跳出工作流去查阅。本系统通过 RAG（检索增强生成）架构，将 17 篇医学文档向量化存储，当医生在 AI 对话中提出医学问题时，Agent 自动触发语义检索，将相关文献片段作为上下文注入 LLM 回答中，实现带源引用的循证医学问答。

### 痛点 4：病历表单填写繁琐

传统病历系统使用大量 JSON 格式自由文本框录入结构化数据（如用药方案、生活建议、专科医师列表），容易填错且校验失败率高。本系统实现了三种可视化结构化输入组件：
- **tags 标签输入**：字符串数组（症状列表、处置项目等）
- **kvlist 键值列表**：动态键值对（病史详情、随访计划等）
- **rows 动态行**：带子 schema 的表格行（用药方案、生活建议、专科医师等）

用户在可视化组件中填写后，系统自动转换为正确的 JSON 格式存入数据库。

## 二、核心逻辑流

### 整体架构：单 Agent 多工具编排

本项目采用**单一 Agent + 多工具调度**架构，非多 Agent 协作模式。Agent 作为中央编排器，根据意图识别结果将任务分发到不同工具执行。选择该架构的原因是医疗场景对延迟敏感，单 Agent 调度避免了多 Agent 间的通信开销和不确定性。

```
用户输入 (自然语言)
        │
        ▼
┌───────────────────────────────┐
│     AgentManager.execute()    │  ← 中央编排器
│                               │
│  ┌─────────────────────────┐  │
│  │ Step 1: 意图识别         │  │
│  │ LLM 5 路分类 + 关键词兜底 │  │  ← 短链推理（单次 LLM 调用）
│  └───────────┬─────────────┘  │
│              │                 │
│     ┌────────┼────────┐       │
│     ▼        ▼        ▼       │
│  database medical  vector     │
│  _query   _chat   _search     │
│     │        │        │       │
│     │        │        ▼       │
│     │        │   ┌─────────┐  │
│     │        │   │ 向量检索  │  │
│     │        │   │ 余弦相似度 │  │
│     │        │   └────┬────┘  │
│     │        │        │       │
│     │        └────────┤       │
│     │          上下文注入     │
│     │                 │       │
│     ▼                 ▼       │
│  ┌─────────────────────────┐  │
│  │ Step 2: 工具执行         │  │
│  │                         │  │
│  │ databaseQuery (3步)     │  │  ← 长链推理
│  │  LLM→查询计划→执行→格式化 │  │
│  │                         │  │
│  │ medicalChat (RAG增强)   │  │
│  │  医学系统提示词+上下文    │  │
│  └───────────┬─────────────┘  │
│              │                 │
│              ▼                 │
│  ┌─────────────────────────┐  │
│  │ Step 3: 结果持久化       │  │
│  │ 保存到 Message 表        │  │
│  │ 更新会话历史 (内存)      │  │
│  └─────────────────────────┘  │
└───────────────────────────────┘
        │
        ▼
  SSE / JSON 响应 → 前端渲染
```

### 长链推理：Database Query Tool 的三步 LLM 流水线

`databaseQueryTool` 是本系统中推理链最长的工具（1028 行），实现了**三步串行 LLM 调用**：

```
用户自然语言
    │
    ▼
┌─────────────────────┐
│ LLM Call 1: 解析意图 │
│ 输入: "最近一周心内科收了多少高血压患者？"
│ 输出: {
│   queryType: "count",
│   filters: [
│     { field: "diagnosisName", op: "contains", value: "高血压" },
│     { field: "createdAt", op: "gte", value: "7天前" }
│   ],
│   groupBy: null,
│   ...
│ }
└────────┬────────────┘
         │
         ▼
┌─────────────────────┐
│ executeQueryPlan()  │  ← 无 LLM 参与，确定性执行
│ 映射到 Prisma 查询   │
│ prisma.diagnosis     │
│   .count({ where })  │
│ → 原始结果: 15       │
└────────┬────────────┘
         │
         ▼
┌─────────────────────┐
│ LLM Call 2: 格式化  │
│ 输入: 原始查询结果   │
│ 输出: "根据系统统计， │
│ 最近7天内共有 15 名  │
│ 高血压相关患者入院…" │
└─────────────────────┘
```

该流程体现了**长链推理**特征：前一步 LLM 的输出是下一步的输入，任何一步的输出偏差都会影响最终结果。

### RAG 长链：向量检索 + 上下文增强

当用户的问题涉及专业知识时，Agent 走 `vector_search` 路径，触发 4 步链式处理：

```
用户: "2型糖尿病的诊断标准是什么？"
    │
    ▼
意问识别 → vector_search
    │
    ▼
┌────────────────────────────┐
│ Step 1: 生成查询嵌入向量     │
│ Aliyun text-embedding-v2   │
└───────────┬────────────────┘
            │
            ▼
┌────────────────────────────┐
│ Step 2: 余弦相似度检索       │
│ 遍历 VectorStore 表所有向量  │
│ 筛选 similarity > 0.7       │
│ 同源文档块聚合               │
└───────────┬────────────────┘
            │
            ▼
┌────────────────────────────┐
│ Step 3: 构建 RAG 上下文      │
│ 将检索结果注入系统提示词      │
│ "请参考以下医学文献回答问题…" │
└───────────┬────────────────┘
            │
            ▼
┌────────────────────────────┐
│ Step 4: medicalChat LLM    │
│ 结合文献上下文生成循证答案   │
│ 标注引用来源                 │
└────────────────────────────┘
```

### AI 临床分析：长上下文推理

患者综合分析是另一个长链场景 —— 将患者的 9 类关联数据格式化为结构化文本，单次 LLM 调用的上下文可超过 2000 字符，要求 LLM 从海量病历信息中提取关键信号并作出临床判断：

```
患者全量数据（基本信息 + 病史 + 症状 + ...）
    │
    ▼
formatPatientContext() → 结构化文本 (2000+ chars)
    │
    ▼
LLM System Prompt（资深临床医学专家角色 + 风险评估标准）
    │
    ▼
LLM 推理输出（结构化 JSON）
{
  "riskLevel": "high/moderate/low",
  "riskSummary": "...",
  "keyFindings": [...],
  "attentionPoints": [...],
  "diagnosisAssessment": "...",
  "treatmentReview": "...",
  "followUpSuggestions": "..."
}
    │
    ▼
持久化到 Patient.aiNotes（分析/建议独立存储）
```

## 三、技术架构总结

| 维度 | 方案 |
|------|------|
| Agent 模式 | 单 Agent 多工具调度（非 LangChain AgentExecutor） |
| 推理链长度 | databaseQuery 最多 3 步串行 LLM 调用；RAG 最多 4 步链式处理 |
| 多 Agent 协作 | 否，采用单一编排器 + 工具分发模式 |
| LLM 双模型策略 | KIMI（推理/分析）+ MIMO（文档生成含联网搜索） |
| 向量存储 | 自研 JSONB 方案（PostgreSQL），JS 余弦相似度 |
| 会话记忆 | 内存级 session history（20 条上限） |
| 结构化输出 | Zod schema 约束工具输入，JSON mode 约束分析结果 |
